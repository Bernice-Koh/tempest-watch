# Conventions

Engineering conventions for tempest-watch. Every rule here exists because the alternative caused pain in some past project. Language-level code style lives in [CODE_STYLE.md](CODE_STYLE.md); this file is about *how we work*, not how we write code.

This project is built lean by design: a Python/FastAPI backend, a React + Three.js frontend, PostgreSQL, a LangGraph-orchestrated matcher calling the LLM endpoint directly, and manual deployment to a single box. No infrastructure-as-code, no managed orchestration. The conventions below assume that shape.

## Folder structure

Three concerns at the root: `backend/`, `frontend/`, `deploy/`. Never mix. `deploy/` holds `docker-compose.yml`, systemd units, and deploy scripts — operational glue, not infrastructure-as-code; this project deploys by hand to a single box on purpose.

Inside `backend/src/`, separate by what code is *for* — `api/`, `services/`, `sources/`, `agents/`, `models/`, `jobs/`, `db/`, `core/` — not by file type. Tests live in a parallel tree mirroring `src/`, not co-located, so test discovery is trivial and prod imports never leak test code.

Create folders when there's code to put in them, not before. Empty directories with `.gitkeep` markers telegraph that someone gave up.

## Dependencies and tooling

- **Python:** managed with [`uv`](https://docs.astral.sh/uv/). Project metadata and deps in `backend/pyproject.toml`; lockfile is `backend/uv.lock` and is committed. Use `uv add <pkg>` to add deps and `uv run <cmd>` to run anything in the project environment.
- **Frontend:** managed with `npm`. Lockfile is `frontend/package-lock.json` and is committed.
- **Python lint/format:** `ruff` for both. Run `uv run ruff check` and `uv run ruff format`.
- **Python types:** `mypy --strict` against `backend/src/`.
- **Frontend lint/format:** `eslint` + `prettier`. Run `npm run lint` and `npm run format`.
- **Pre-commit:** the `pre-commit` framework runs the above before every commit. Config in `.pre-commit-config.yaml`.
- **Python version:** 3.13. Pinned via `backend/.python-version` and `pyproject.toml`. Bump only when the whole dep tree supports it.

## Database, ORM, migrations

- **Engine:** PostgreSQL.
- **Production:** AWS RDS managed Postgres. Backups, patching, monitoring are RDS's job, not mine.
- **Local dev and tests:** Postgres in `deploy/docker-compose.yml`, started with `docker compose up -d postgres`. Tests run against this container, not RDS — tests must work offline.
- **ORM:** SQLAlchemy 2.x with the typed declarative API (`Mapped[...]`, `mapped_column(...)`). Pydantic models are separate from ORM models; never use one for both roles.
- **Migrations:** Alembic. One revision per schema change. Autogenerated revisions must be read and edited before committing — `autogenerate` is a starting point, not the final answer. Apply with `alembic upgrade head`.
- **Schema source of truth:** the migration history, not the model classes. Anything not in a migration doesn't exist.

## Testing

- `pytest` for Python, `vitest` for the frontend (Vite-native).
- Mirror the `src/` layout in `tests/`. A test file's path tells you what it covers.
- One behaviour per test. Multiple assertions are fine when they verify the same behaviour; bad when they hide unrelated checks.
- Tests run migrations against the local docker-compose Postgres — never against prod, never SQLite as a stand-in. The pattern is: spin up a clean DB, run all migrations, run the test suite, tear down. This catches migration bugs the same way prod will.
- Mock external HTTP (eBay, PriceCharting, Carousell, the LLM API). Never hit them from tests.
- Keep a small labelled set of real listings with known-correct matches, and assert the matcher's precision/recall against it. Each pipeline stage's impact should be measured, not assumed.

## Logging

- Structured JSON via `structlog`. Never `print`.
- Every log line carries a correlation ID propagated through the request or job run.
- `DEBUG` for tracing, `INFO` for normal flow, `WARNING` for unexpected-but-recoverable, `ERROR` for failures. `CRITICAL` is reserved for "page someone" scenarios.

## Error handling

- API routes catch only at the handler boundary; scheduled jobs catch only at their entrypoint. Internal code raises typed exceptions from `core/errors.py`.
- Route handlers map exceptions to HTTP responses in one place. Business logic stays free of HTTP concerns and trivially unit-testable.
- Never swallow exceptions silently. If you catch and continue, log at `WARNING` minimum with the exception attached.

## Scheduled jobs

- Scheduled work (daily price snapshots, source polls) is run by **systemd timers** on the box, calling Python entrypoints in `backend/src/jobs/`. Not APScheduler-in-FastAPI — keeps the web process and the cron-style work decoupled so one can fail without taking the other down.
- Each job is a thin entrypoint that does setup (DB session, logging context, correlation ID) and calls a service. The service is where the work and the tests live.

## Git

- Commit messages follow Conventional Commits: `feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`, `ci:`.
- Subject line ≤ 72 chars, imperative mood ("add X", not "added X" or "adds X").
- Body explains *why* if it's not obvious from the diff. Skip the body for trivial commits.
- One concern per commit. If "and" wants to appear in the subject, split the commit.

## Pull requests

- Branch names: `feat/<short-description>`, `fix/<...>`, `chore/<...>`.
- PR description covers what changed, why, and how to test. Two sentences is often enough.
- CI must pass before merge. No exceptions, even for "obvious" hotfixes.
- Squash on merge to keep main's history linear.

## Secrets

- Never commit secrets, even temporarily, even "I'll rotate it later".
- Secrets live in `.env` on the box, never in git, with permissions locked down (`chmod 600 .env`).
- `.env.example` (template only) is committed. `.env` and `.env.local` are not.
- If this outgrows a flat `.env`, reach for encrypted-at-rest secrets (e.g. SOPS + age) before a managed secrets service — keep the deploy lean.
